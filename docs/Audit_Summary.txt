1. BACKEND ARCHITECTURE -- Grade: B
--------------------------------------------------------------------------------

  Strengths:
  - Clean service layer pattern with 37 services across core, ML, and PC agent
  - Async/await used correctly throughout; blocking calls wrapped in
    asyncio.to_thread()
  - WebSocket + polling hybrid catches external changes (Alexa, Hue app)
  - Mode-change callback system is elegant and extensible
  - SQLAlchemy 2.0 async ORM with proper parameterized queries
  - ML services follow clean embedded pattern with graceful degradation --
    every ML feature has a non-ML fallback
  - Rule engine adds learned time-based patterns from event data
  - Event logging system is comprehensive (6 tables, fire-and-forget pattern)
  - NWS weather migration removed paid API dependency
  - WiFi presence detection replacing Alexa geofence
  - Rate limiting via slowapi (120 req/min default, 10/min on TTS)

  Issues:
  - automation_engine.py is 1,944 lines (grew from 1,825 in one day) --
    urgently needs splitting into mode logic, light state calculator, and
    effect manager. The file keeps absorbing every new capability
  - main.py lifespan function is ~350 lines initializing 20+ services --
    extract to a bootstrap.py or service registry
  - Service dependency injection via direct attribute assignment at
    main.py:314-328 (automation._confidence_fusion = ..., _behavioral_predictor
    = ..., rule_engine._fusion = ...). Invisible coupling; not discoverable
    from __init__ signatures. Refactor to constructor args
  - Incomplete shutdown in lifespan finally block: only hue_v2, rec_service,
    and camera get close() called. Sonos, TTS, music_mapper, automation,
    scheduler, rule_engine, presence, ambient_sound, pihole, bar, and plant
    services leak open sockets/sessions on shutdown
  - Event logger silent-drop pattern: event_logger.py:83-84, :146, :174,
    :205 catch bare Exception and log-only. ML training data gaps accumulate
    invisibly -- no retry queue, no alert on repeated failures
  - Per-light manual override expiration not enforced in code. Mode-level
    4h timeout works (automation_engine.py:1552), but _manual_light_overrides
    dict only clears via clear_override() -- no per-entry timeout
  - Scene override silent fallback hole at automation_engine.py:1034-1045:
    if both v2 scene and preset fallback fail, lights stay in prior state
    with no ERROR-level log
  - scenes.py route file is 768 lines with 20 preset definitions inline --
    extract presets to a separate module
  - Light IDs are strings in some places, ints in others -- inconsistent typing
  - 22+ bare "except Exception" clauses across the backend -- should catch
    specific exceptions on critical paths
  - No middleware for request logging/tracing
  - No circuit breaker for Hue/Sonos API calls (if bridge hangs, polling blocks)

--------------------------------------------------------------------------------

  Strengths:
  - Clean Svelte store pattern with WebSocket dispatch (12 stores)
  - 36 components with clear single-responsibility
  - 4 routes: Home, Music, Analytics, Settings
  - Analytics page is now a live decision pipeline dashboard -- SVG
    confidence ring, 5 animated signal cards, collapsible historical
    analytics. Real-time fusion scores update on every pipeline broadcast
  - Generative canvas is visually distinctive -- not generic AI dashboard
  - Good idle detection (60s timeout, auto-hide UI elements)
  - WebSocket reconnection with build_id auto-reload is clever
  - Playing-card deck for mode selection is unique
  - ModeSuggestionToast surfaces rule engine predictions to user

  Issues:
  - Memory leaks: GenerativeCanvas.svelte -- three store subscriptions
    cleanup may not catch all paths
  - Silent API failures: api.js -- .catch(() => ({})) swallows all errors,
    UI never shows failure state
  - api.js has no AbortController/timeout on apiGet/apiPost -- slow
    backend hangs fetches indefinitely
  - WebSocket unknown-type messages silently dropped (init.js:68-114 has
    no `default` case in the type switch)
  - ws.js build_id reload calls window.location.reload() unconditionally --
    no error path if the reload itself fails during a deploy race
  - ParallaxScene.svelte rAF loop has no FPS cap (unlike GenerativeCanvas
    which caps at 15fps); paints ~9x as much work on a 144Hz display
  - Modal portal duplication risk: PlantWidget and BarWidget each append
    modals to document.body via a local portal action with no singleton
    guard. Two-mount = duplicated listeners
  - ErrorToast.svelte lacks role="alert" / aria-live -- screen readers
    silently miss the toast
  - LightCard.svelte:11-19 redefines color presets already in
    theme.js:147-156. Deduplicate to prevent drift
  - Accessibility gaps:
    * Empty alt text on album art images
    * Slider missing aria-labelledby
    * --text-muted color (#5c5e6a on #08080c) fails WCAG AA (~2.5:1)
    * No <html lang="en">
  - No component tests or E2E tests
  - Three.js (~500KB) + Threlte (~100KB) loaded unconditionally, even when
    MoonScene (sleeping mode) isn't active
  - No prefers-reduced-motion gate on heavy animations

4. SECURITY -- Grade: C (was D)
--------------------------------------------------------------------------------

  FIXED since last audit:
  - CORS locked to specific LAN origins (was allow_origins=["*"])
  - Rate limiting via slowapi on all endpoints
  - WebSocket JSON parsing wrapped in try-catch with validation

  REMAINING CRITICAL:
  - Zero authentication: Every API endpoint is fully open. Anyone on WiFi
    can control lights, play audio, upload files, read all data. All 80+
    endpoints, including ML model promotion/demotion, camera enable/disable,
    and rule engine management are unprotected.
  - File upload: music.py -- No file size limit, no content-type validation,
    no XXE protection on XML parsing (Apple Music import).

  HIGH:
  - Preview URL injection: music.py -- preview_url passed directly to
    sonos.play_uri() with no URL validation (SSRF risk).
  - Plant App sends email + password in JSON body over HTTP (not HTTPS).

  MEDIUM:
  - MCP query_db accepts arbitrary SELECT including UNION attacks.
  - CORS origins list exposes LAN topology (192.168.1.x addresses).
  - Camera service has no auth -- anyone on LAN can enable/disable face
    detection via POST /api/camera/enable.
  - HTTP only: No TLS. Acceptable for LAN-only but blocks Cloudflare Tunnel.

  Verified safe:
  - .env IS in .gitignore -- secrets not committed (verified across all
    171+ commits in git history, zero matches)
  - .env.example uses only placeholder values -- safe on GitHub
  - No hardcoded secrets in any Python or JavaScript source files
  - subprocess.run() uses list args (no shell injection)
  - SQL in route files uses parameterized :placeholder syntax
  - data/ directory (SQLite DB, ML models) fully gitignored
  - *.png gitignored (except frontend background assets)
  - .mcp.json contains only command definition, no secrets
  - ML model files contain only learned weights, no PII

5. MACHINE LEARNING / INTELLIGENCE -- Grade: B+ (NEW)
--------------------------------------------------------------------------------

  Architecture:
  - 8 ML services in backend/services/ml/ (2,093 LOC total)
  - All services follow shadow-before-promote pattern
  - Local-only processing -- no cloud APIs for inference
  - Every ML feature has a non-ML fallback (graceful degradation)
  - Confidence fusion: 5-signal weighted ensemble for mode decisions
  - Nightly retraining at 4 AM via AsyncScheduler
  - Auto-demotion: accuracy < baseline for 3 consecutive nights

  Services:
  - ConfidenceFusion (228 LOC) -- weighted ensemble of process, camera,
    audio, predictor, and rule engine signals. Auto-apply at 95%+ when
    idle/away, override stale processes at 98%+ with 80%+ agreement.
    Default weights with nightly accuracy-driven updates
  - ModelManager (193 LOC) -- model persistence, versioning, metadata
  - BehavioralPredictor (327 LOC) -- LightGBM mode prediction from 10
    temporal + behavioral features. Shadow mode, needs 500+ events to train
  - LightingPreferenceLearner (208 LOC) -- EMA (alpha=0.3) per-light
    preferences from manual adjustments. Needs 5+ adjustments per combo
  - AudioClassifier (504 LOC) -- YAMNet (TFLite) 521 AudioSet classes
    mapped to 9 Home Hub classes. Shadow mode, 2-3s inference cycle
  - MusicBandit (260 LOC) -- Thompson sampling playlist selection with
    Beta(alpha, beta) distributions per arm. Active, exploring
  - FeatureBuilder (202 LOC) -- temporal + behavioral feature engineering
  - MLDecisionLogger (170 LOC) -- decision logging with reasoning chains,
    backfills actual_mode on next transition

  Additional ML-adjacent:
  - CameraService (371 LOC) -- MediaPipe face detection, 15s away detection,
    opt-in via camera_enabled setting, frames never persisted
  - Smart screen sync -- K-means (n=5) dominant color extraction, replaces
    naive pixel averaging
  - RuleEngineService (295 LOC) -- frequency-based time patterns from events

  Privacy:
  - Camera frames: numpy arrays in memory only, overwritten each 5s cycle.
    Never serialized, persisted, networked, or logged
  - Audio: raw PCM overwritten each 1s read cycle. Mel-spectrograms are
    one-way (cannot invert to intelligible audio). Only labels persist
  - Models: aggregate statistical patterns only. No PII reconstructable
  - Full ML data wipe available from Settings (models + decisions + metrics)
  - 90-day rolling retention on event tables

  Model files (all in data/models/, gitignored):
  - yamnet.tflite (16 MB) -- auto-downloaded from Google GCS
  - yamnet_class_map.csv (14 KB) -- auto-downloaded
  - blaze_face_short_range.tflite (225 KB) -- auto-downloaded
  - mode_predictor.lgb -- created after 500+ events (pending)
  - lighting_prefs.json -- created after 5+ adjustments per combo (pending)
  - music_bandit.json -- created on first playlist selection
  - model_meta.json -- tracks version, accuracy, status per model

  Issues:
  - ML test coverage still patchy: audio_classifier (26) + camera_service
    (16) have tests. behavioral_predictor, lighting_learner, music_bandit,
    feature_builder, ml_logger, confidence_fusion, rule_engine do NOT
  - BehavioralPredictor in cold start period -- needs 500+ activity events
    (~1 week of data) before first training
  - AudioClassifier still in shadow mode -- needs accuracy comparison with
    existing RMS detector before promotion
  - ConfidenceFusion wired into automation_engine.py:1659-1661 but fusion
    *weights* still hardcoded -- no nightly accuracy-driven update. The
    promised weight-learning step from ml_metrics is not shipped
  - Lighting-learner overlay applied silently: automation_engine.py:1077-1083
    applies the LightingPreferenceLearner delta to the target state but no
    ml_decisions row is written for the overlay. Hard to tell from logs
    whether the learner is actually touching a light on any given cycle
  - No A/B testing framework for comparing ML vs rule engine decisions
  - lightgbm is optional dependency -- silent graceful degradation means
    predictor failures could go unnoticed without monitoring

6. PERFORMANCE -- Grade: B-
--------------------------------------------------------------------------------

  Issues:
  - Hue polling at 1s intervals -- aggressive for 6 bulbs. 3-5s would work
  - Sonos polling at 2s intervals -- could be 5s
  - Individual WebSocket broadcasts per light on scene activation (N sends
    instead of batched)
  - K-means screen sync is heavier than naive averaging (~10ms per frame)
    but produces significantly better color selection
  - MediaPipe face detection adds CPU load (~2%, mitigated by opt-in toggle
    and 5s capture interval)
  - LightGrid.svelte runs Object.values($lights).sort() on every reactive
    change -- should memoize
  - GenerativeCanvas particle count resizes array frame-by-frame during
    transitions (GC pressure)
  - MoonScene shader materials at module scope -- VRAM leak risk on
    unmount/remount
  - Three.js + Threlte loaded unconditionally (~600KB) even when sleeping
    mode isn't active

  ML performance budget (sustained):
  - Audio classifier: 5% CPU, 50-200MB RAM, every 2-3s
  - Camera service: 2% CPU, 50-100MB RAM, every 5s
  - Behavioral predictor: <1% CPU, <10MB RAM, every 60s
  - Music bandit: <1% CPU, <1KB RAM, on mode change
  - Lighting learner: <1% CPU, <1MB RAM, on mode change
  - Total ML overhead: <10% CPU, 200-400MB RAM

  OK:
  - 15fps cap on GenerativeCanvas is appropriate
  - asyncio.to_thread() correctly used for blocking Hue bridge calls
  - SQLite is fine for current data volume (~184KB DB)
  - Music bandit is O(1) per decision -- negligible overhead

  MACHINE LEARNING IDEAS (new, April 16)
  -----------------------------------------------------------------------

  Override Reason Classifier with Soft Counterfactuals        [Medium]
    Train a lightweight sequence classifier on the 90s of sensor state
    leading up to every manual override (time-of-day, prior 3 modes,
    weather, last playback, camera presence, ambient audio). Model output
    is a cluster label for the *reason* the user overrode: "too bright
    for evening screen time", "relax-mode picked wrong music for my mood",
    "winddown too early while guests present". On the NEXT occurrence of
    a matching context, do NOT auto-apply -- surface a soft counterfactual
    toast: "Last Tuesday at 9pm raining, you switched working -> relax.
    Try that now, or stay working?" Distinct from Mood Drift Detection
    (which tracks multi-day *derivative* of preferences); this tracks the
    *reason* for single overrides and drives surgical interventions.
    Touches: new override_reason_classifier.py, ml_logger.py,
    ModeSuggestionToast.svelte, confidence_fusion.py (new signal lane)

  Per-User Transition-Curve Preference Learning               [Medium]
    Learn not just WHICH mode, but HOW each mode should transition. Do
    you prefer bri-first-then-color, crossfade, or snap? At what speed?
    Data source: mid-transition manual light adjustments already captured
    in light_adjustments. If the user consistently nudges brightness up
    during a 4s crossfade (they keep wanting it brighter sooner), the
    model learns to lead with brightness next time. Autonomously updates
    MODE_TRANSITION_TIME and transition *order* per (mode, time-of-day)
    bucket. Graceful degradation: falls back to hardcoded defaults when
    N < 20 transitions observed for that bucket. All existing ML features
    personalize the destination *state*; this personalizes the *trajectory*.
    Touches: new transition_curve_learner.py, automation_engine.py
    (_apply_state pre-hook), new transition_preferences column


  GENERAL HOME HUB IDEAS (new, April 16)
  -----------------------------------------------------------------------

  Focus Envelope                                              [Medium]
    A pomodoro-by-lights mode. During focus sessions, the ambient lights
    smoothly constrict toward desk-dominant (kitchen fades, L1 dims, L2
    desk stays solid); during breaks, they diffuse back outward. The
    brightness envelope itself IS the pomodoro timer -- no distracting
    clock, no sound, no overlay. Auto-triggers when Cursor / VSCode /
    `claude` is focused for >25 min with no alt-tab >18 min. Break
    envelope: 5 min diffuse + ambient noise shift. User sets session:break
    ratio. Distinct from existing `working` mode (which is static) and
    from Adaptive Transition Choreography in Future_Development.md (which
    is a staggered wave on mode *change*): Focus Envelope is a continuous
    low-amplitude modulation *during* a mode, with semantic meaning.
    Touches: new focus_envelope.py, Settings toggle, FocusChip.svelte

  Apartment Logbook -- Nightly Auto-Journal                   [Easy]
    A silent nightly (2am) job writes a single-file Markdown journal entry
    summarizing the day in narrative prose: "Worked 4h12m (9:14am-2:05pm,
    paused at 11:30 for kitchen). Gaming 1h48m. Rain rolled in at 2:14pm,
    which triggered candle effect twice and shifted winddown 12 min
    early. Overrode the winddown routine once (stayed on watching until
    11:52pm)." Pure read over existing event_logger tables -- no new
    data sources. Writes to data/journal/YYYY-MM-DD.md. Surfaced behind
    a /journal route (hidden from main nav). Also primes any future
    LLM-backed features: the journal file is ready-made grounding context.
    Distinct from Dashboard Replay (visual scrubbing) and Sleep Analytics
    (sleep-focused) -- this is a compact narrative summary of the whole
    day, with search-friendly prose.
    Touches: new journal_service.py (scheduled at 2am), /journal route
